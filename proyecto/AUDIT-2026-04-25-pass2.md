Second audit, focused on surfaces the first pass did not cover.
Three parallel agents:

1. **Broker** (`crates/broker/`) — NATS reconnect, disk queue,
   DLQ, local broker, topic matching, circuit breaker.
2. **Admin HTTP** (`src/main.rs::run_admin_server` + handlers) —

## 🔴 Critical — fix before next ship

### B-C1. NATS subscription dies silently on reconnect
- `crates/broker/src/nats.rs:170-201`
- A subscription's read task holds the original `nats_sub`. When
  NATS drops, the task keeps polling a dead stream. After
  reconnect, the task is still alive but receives nothing. New
  publishes go to disk queue and drain to NATS — but the original
  subscriber's handler never sees them. A subscriber created
  before the disconnect is permanently muted until the process
  restarts.
- **Fix:** thread a cancellation token into each subscription
  task, exit on disconnect, and have the broker re-subscribe
  automatically when state monitor reports `Connected`. Or
  document that subscriptions don't survive reconnects and require
  a fresh `subscribe()` call.

### B-C2. Disk queue: publish-then-delete is not atomic → duplicate delivery on crash
- `crates/broker/src/disk_queue.rs:153-159`
- Drain pulls a row, publishes to NATS, then deletes the row.
  If the delete fails (lock, disk error, crash), the event is now
  on NATS *and* still in the queue. Next drain re-publishes it.
- **Fix:** wrap the per-row drain in a SQLite transaction:
  `BEGIN`, mark in-flight (or delete pre-publish), publish,
  `COMMIT` or rollback. Pair with idempotency on the consumer
  side if exactly-once is wanted.

### B-C3. Circuit breaker can stick HalfOpen forever
- `crates/broker/src/nats.rs:51-59` (cfg `failure_threshold:1, success_threshold:1`)
- After backoff expires, the breaker transitions to HalfOpen
  but only flips to Closed on the next *real* publish. If no one
  publishes for hours after recovery, the breaker stays HalfOpen
  and rejects when traffic returns. Doubling backoff in
  back-to-back HalfOpen→Open also extends the dead window.
- **Fix:** raise `failure_threshold` and `success_threshold` to
  2; or spawn a periodic no-op probe that exercises the breaker.

### B-C4. DLQ promotion race: row in both `pending_events` and `dead_letters`
- `crates/broker/src/disk_queue.rs:162-207`
- Increment-attempts and move-to-DLQ run as separate statements.
  If the INSERT into `dead_letters` succeeds but the DELETE from
  `pending_events` fails (or the process crashes between them),
  the same event lives in both tables. Next drain re-processes
  it from `pending_events` and replays it to NATS.
- **Fix:** single transaction with a `CASE WHEN attempts >= 3`
  branch that does INSERT + DELETE atomically, or run the DLQ
  move under a serializable transaction with retry.

### A-C1. No file lock on YAML mutations from admin handlers
- `src/main.rs:3822, 3935, 4102` (add/edit/delete telegram channel)
- Each handler reads the YAML, mutates in memory, writes back.
  Concurrent admin requests can interleave and silently lose
  writes (read A → read B → write A → write B drops A's change).
- **Fix:** per-file `std::sync::Mutex<()>` guarded by path,
  or atomic write-rename pattern (`tmpfile + fs::rename`) plus
  serialise per-file via a tokio mutex held in the admin server
  state.

## 🟠 High — silent data loss or persistent state corruption

### B-H1. Local broker drops messages on slow consumer, returns Ok
- `crates/broker/src/local.rs:41-51`
- Channel-full → `try_send` errors, broker swallows the error
  and returns `Ok(())` to the publisher. Slow subscriber → silent
  message loss, publisher doesn't know.
- **Fix:** at minimum bump a counter, log at warn (with
  rate-limit), and consider returning a structured
  `BrokerError::SubscriberLagging` so the publisher can decide.

### B-H2. Backpressure deletes can race
- `crates/broker/src/disk_queue.rs:88-100`
- At hard cap, both concurrent enqueues sleep then DELETE the
  oldest row → two rows dropped instead of one. Backpressure
  intent partially defeated.
- **Fix:** serialise enqueues with a `Mutex` or rely on a single
  SQLite `IMMEDIATE` transaction.

### A-H1. Admin password printed to stdout
- `src/main.rs:2562`
- The random per-launch admin password is `println!`ed. In any
  containerised, journald-aggregated, or CI-captured deployment,
  the password ends up in long-lived logs.
- **Fix:** print only a fingerprint (first/last 4 chars or sha256
  prefix), and log the real password through a write-once channel
  the operator pulls actively (e.g., a unix socket, a temp file
  at `/run/agent/admin.password` chmod 600).

### A-H2. Cookie not invalidated server-side on logout
- `src/main.rs:2608, 2708`
- `validate_cookie` checks expiry but there is no server-side
  revocation list. A captured cookie (browser history, RAM dump,
  shared workstation) remains valid for the full 24h TTL after
  logout.
- **Fix:** keep an in-memory `HashSet<u64>` of issued+still-valid
  expirys (cookie embeds expiry as the unique part); remove on
  logout; prune periodically.

### MCP-H1. SSE stream parser skips non-UTF8 chunks → tool replies lost silently
- `crates/mcp/src/http/sse.rs:39-44`
- A non-UTF8 byte in the stream → `continue`. The frame parser
  loses sync; subsequent messages are misaligned. Pending tool
  calls time out without explanation.
- **Fix:** on UTF8 error, break and surface a transport error so
  the caller can reconnect / retry instead of waiting on a
  wedged stream.

### MCP-H2. SSE inflight removed before response can land
- `crates/mcp/src/http/client.rs:281-336`
- If `post_sse_raw` errors before sending, the inflight slot is
  cleared. If the request actually reached the server (in-flight
  packet) and gets processed, the response lands but no slot is
  waiting — orphaned. Next call reuses the ID and receives a
  stale response from the previous request.
- **Fix:** keep the slot until either response arrives, timeout
  fires, or session is torn down — never on ambiguous send error.

### MCP-H3. Child process death race vs pending requests
- `crates/mcp/src/client.rs:157-165`
- If the child exits before the supervisor task is scheduled,
  `request()` callers may insert into `pending` after
  `drain_pending()` has already run, hanging until timeout.
- **Fix:** `Arc<AtomicBool>` "child_dead" flag, checked at the
  start of `request()` (return error early); set by the
  supervisor before `drain_pending()`.

### WA-H1. Empty `outbound_allowlist` = unrestricted, no warning
- `crates/plugins/whatsapp/src/tool.rs:41-58`
- `is_empty()` returns "allowed" for back-compat. An operator
  migrating from older config without an allowlist gets
  unrestricted outbound — a prompt injection can DM any number.
- **Fix:** log at boot when the allowlist is empty
  (`tracing::warn!("WhatsApp outbound allowlist empty for agent {agent_id} — unrestricted sends enabled")`).
  Optionally fail-closed under a new config flag.

### WA-H2. Outbound dispatcher blocks on slow URL fetches
- `crates/plugins/whatsapp/src/dispatch.rs:196-229`
- The dispatcher inline-fetches media URLs with a 60s timeout.
  A slow URL stalls every other outbound message for that
  account for up to 60s.
- **Fix:** push media downloads onto a per-channel
  `tokio::sync::Semaphore`-bounded queue, return "queued"
  immediately, dispatch when fetch completes.

## 🟡 Medium — wrong-but-bounded, or hidden-by-default

### B-M1. Multi-row drain not transactional
- `crates/broker/src/disk_queue.rs` (drain loop)
- Per-row publishes happen outside any transaction. If row 50 of
  100 errors, rows 1-49 are committed deleted; rows 50-100 are
  retried next drain — but the in-memory state assumed atomicity.
- **Fix:** wrap full drain pass (or per-batch) in a transaction.

### B-M2. DLQ replay resets attempts to 0; loops forever on persistent error
- `crates/broker/src/disk_queue.rs:236-263`
- A DLQ entry replayed by an operator gets `attempts = 0`. If the
  underlying error (bad topic, malformed payload) is permanent,
  it loops back to DLQ after 3 more failures.
- **Fix:** preserve a `replay_count` field or surface the
  original failure reason in the replay UI so operators can
  decide before re-injection.

### B-M3. Topic matcher accepts empty segments and empty pattern
- `crates/broker/src/topic.rs:7-22`
- `topic_matches("", "")` returns `true`. Leading/trailing dots
  produce empty segments that pass the matcher in surprising
  ways.
- **Fix:** validate topics at `publish`/`subscribe` boundary;
  reject empty strings, leading/trailing dots, empty segments.

### B-M4. State monitor doesn't await drain — silent stall on spawn failure
- `crates/broker/src/nats.rs:99-107`
- Drain is `tokio::spawn`'d on reconnect. If the spawn itself
  fails (very unlikely but possible under task budget pressure),
  the disk queue never drains and operators see no error.
- **Fix:** if spawn fails, leave the breaker open or emit a loud
  error so reconnect doesn't appear successful when nothing is
  flowing.

### A-M1. Error responses leak filesystem paths
- `src/main.rs:3876, 3970, 4116` and similar
- `format!("read {path}: {e}")` includes absolute paths in JSON
  responses. Authenticated attacker maps the install layout from
  a typo or permission error.
- **Fix:** return generic strings (`"config read failed"`) to the
  client, log the full path via `tracing` only.

### A-M2. SPA path-traversal mitigation depends on URL-decoding order
- `src/main.rs:4512-4527`
- The `..`-segment check runs after `dir.join(rel)`. If the
  caller hands raw, undecoded `%2e%2e/etc/passwd`, the check
  passes because the literal path no longer contains `..`.
- **Fix:** decode before `split('/')`, or reject any `%` in the
  asset path entirely (admin SPA shouldn't need encoded paths).

### A-M3. Tunnel `Drop` doesn't await the kill
- `crates/tunnel/src/lib.rs:60-69`
- `Drop::drop` triggers `start_kill` but cannot `await`. On
  abnormal exit, `cloudflared` lingers; the tunnel URL routes to
  a dead loopback. With a stale URL in browser history an
  attacker can hit a partially-up admin.
- **Fix:** add an explicit `drop_guard` watchdog: a separate
  thread that polls the parent PID and sends SIGKILL to
  cloudflared if the parent exits without calling `shutdown`.

### A-M4. Telegram token probe has no SSRF guards
- `src/main.rs:3822` (`add_telegram_channel` token validation)
- The token string could be coerced into an HTTP target if the
  validation path ever interpolates it into a URL. Today it
  flows to the official Telegram API path, but the input is not
  shape-checked beyond non-empty.
- **Fix:** require `^\d+:[A-Za-z0-9_-]{30,}$` (Telegram's bot
  token shape) before any network use.

### MCP-M1. Tool name collisions silently first-wins
- `crates/core/src/agent/mcp_catalog.rs:124-133`
- When two servers declare the same prefixed tool, the first
  registered wins; the second is dropped with a warning. No
  per-server priority knob.
- **Fix:** allow `mcp.<server>.priority` config; log which
  server's tool wins.

### MCP-M2. `mcp_server_bridge`: `expose_proxies=true` exposes ext_/mcp_ proxy tools
- `crates/core/src/agent/mcp_server_bridge.rs:85-112`
- When the agent runs as an MCP server, the flag intended for
  ergonomics also exposes internal proxy tools meant to delegate
  to *other* MCP servers — effectively turning the agent into a
  router for any client that connects.
- **Fix:** always filter proxy tools regardless of the flag;
  scope `expose_proxies` to non-proxy categories only.

### MCP-M3. SSE block parse failures silently skipped
- `crates/mcp/src/http/sse.rs:85-134`
- A malformed JSON-RPC block is logged at warn and skipped. If
  it carried a tool result, the caller times out.
- **Fix:** count consecutive parse failures; after N (e.g., 5),
  break the stream and surface a transport error.

### WA-M1. Media writes are not atomic
- `crates/plugins/whatsapp/src/media.rs:248`
- `std::fs::write` writes in place; partial writes on crash
  leave a truncated file at a path future code treats as
  "already downloaded".
- **Fix:** write to `<final>.tmp` then `fs::rename` (atomic on
  same FS).

### WA-M2. Daemon-collision check has a race window
- `crates/plugins/whatsapp/src/session.rs:128-136`
- `check_daemon_collision()` then `creds_path.exists()` are not
  atomic — a second process can spawn between the two, or a
  partially-written `creds.json` from a killed pairing flow
  passes the existence check.
- **Fix:** advisory `flock` on `.whatsapp-rs/.lock` covering
  both checks; validate the JSON parses cleanly, not just that
  the file exists.

### WA-M3. Reactive→proactive fallback can drop messages
- `crates/plugins/whatsapp/src/dispatch.rs:143-149`
- After a reactive timeout the slot is cleared. A subsequent
  proactive message on the same session within the timing window
  may be incorrectly treated as already handled.
- **Fix:** keep timeout-fired session ids in a separate set and
  treat them as "no longer reactive" rather than "already sent".

### WA-M4. Inbound media event publishes before download completes
- `crates/plugins/whatsapp/src/bridge.rs:59-97`
- `MediaReceived` event is published immediately; the download
  runs async and may fail without ever delivering the file. The
  agent replies "thanks for the photo", file isn't there.
- **Fix:** delay the inbound event until the download lands, or
  include `download_status: pending|failed` in the event so the
  agent can adapt.

### WA-M5. Session ID does not include the instance label
- `crates/plugins/whatsapp/src/session_id.rs`
- Two accounts (ana, kate) receiving from the same external JID
  hash to the same session id. Cross-instance dedup collisions
  lose messages.
- **Fix:** seed the UUIDv5 with `format!("{instance}.{jid}")`.

### WA-M6. Voice transcription failure silently drops the original message
- `crates/plugins/whatsapp/src/transcriber.rs:59-73`
- Transcribe error → returns `None`. The agent never sees the
  audio at all.
- **Fix:** publish a synthetic text entry
  (`"[audio — transcription failed]"`) so the agent can ask for
  a retry instead of treating the message as never-arrived.

## ❌ Auditor overstatement / not what it looked like

- **A-CSRF (auditor's #3):** the auditor flagged "no CSRF token
  enforcement". The cookie is set with `SameSite=Lax`, and
  modern browsers block cross-site `fetch` with credentials by
  default. The risk is real for older browsers and form-based
  attacks, but it's not "trivially exploitable from any tab".
  Demoted from CRITICAL to a low-priority hardening item.
- **A-Path traversal in DELETE handler:** the auditor claimed an
  attacker could delete arbitrary YAML files. The handler only
  matches the instance string against entries already loaded
  from `config/plugins/<plugin>.yaml`; it never opens
  attacker-controlled paths. Real risk is the *plugin* path
  (`<plugin>` in the URL), already validated against
  `[a-z0-9_-]+` at line 4108-4112. Reduced to "validate
  `instance` for parity".
- **A-Random source:** auditor questioned whether `rand::random`
  is a CSPRNG. It is — `rand` delegates to OS getrandom by
  default. False alarm.
- **MCP-Cookie pool growth:** the cookie jar growth is
  per-`reqwest::Client` instance, and the runtime keeps one
  client per MCP session. The "leak" is bounded by the session
  count, which is small. Real but not urgent.

## Recommended order of attack

If you have one focused day:

1. **B-C2** (publish-delete atomicity) — ~10 lines, eliminates
   duplicate delivery.
2. **B-C4** (DLQ promotion atomicity) — same shape.
3. **B-C1** (NATS sub on reconnect) — needs a cancellation token
   thread-through but otherwise contained.

   tokio mutex; ~20 lines.
5. **A-H1** (password to stdout) — fingerprint or sidechannel.
6. **WA-H1** (empty allowlist warn) — one line.
7. **MCP-H1** (SSE non-UTF8 break) — flip `continue` → `break`
   plus surface a transport error.

Everything else is medium-or-lower and can wait for the next
hardening pass.
