# Phase 77.20.2 Design

## Scope

This document covers only Phase `77.20.2` of the proactive-mode feature:

- Intercept `Sleep { duration_ms, reason }` at runtime.
- Prevent the Sleep sentinel from flowing back into the model as a normal tool result.
- Park the goal inside `driver-loop`.
- Wake the goal later with a synthetic `<tick>` prompt.

Explicitly out of scope for this slice:

- Persistent `Sleeping` state in the registry/store.
- Restore-after-restart behavior.
- Real inbound-channel interruption waking a sleeping goal.
- Daily proactive budget enforcement.
- Final telemetry/docs/admin-ui sync.

Those land in `77.20.3`, `77.20.4`, and `77.20.5`.

## Goal

When the model calls `Sleep`, the current turn must close cleanly, the goal must remain alive in-process, and the orchestrator must resume the goal later with a stable `<tick>` meta prompt instead of replaying the sentinel JSON into normal conversation context.

## Context

The repository already ships the `77.20.1` base:

- `crates/config/src/types/proactive.rs` defines `ProactiveConfig`.
- `crates/core/src/agent/sleep_tool.rs` defines `SleepTool` and the sentinel JSON payload.
- `crates/core/src/agent/proactive_hint.rs` injects the proactive-mode system hint.
- `src/main.rs` registers `Sleep` only when proactive mode is enabled somewhere for the agent.

The missing piece is runtime orchestration in `crates/driver-loop/`.

## External References

Reference behavior comes from `claude-code-leak`:

- `src/tools/SleepTool/prompt.ts`
  - `Sleep` must be preferred over `Bash(sleep ...)`.
  - The user can interrupt sleep.
  - Tick prompts are periodic check-ins.
  - The model should reason about the 5-minute prompt-cache TTL.
- `src/types/textInputTypes.ts`
  - Sleep wake-ups are modeled as system/meta prompts, not as ordinary user text.
- `src/utils/systemPrompt.ts`
  - Proactive mode keeps a stable base prompt and layers agent instructions on top.

Our Rust port keeps the same behavior but fits the existing microservice/driver-loop split.

## Design Summary

Introduce a small proactive runtime inside `crates/driver-loop` with three responsibilities:

1. Translate the Sleep sentinel into a typed driver outcome.
2. Pause the goal for the requested duration with cancellation-aware waiting.
3. Resume the goal by injecting a deterministic `<tick>` prompt into the next turn.

The key structural decision is to use a dedicated outcome in `nexo_driver_types` instead of overloading `AttemptOutcome::Continue`.

Recommended new outcome:

```rust
AttemptOutcome::Sleep {
    duration_ms: u64,
    reason: String,
}
```

This keeps driver control-flow explicit and avoids replay-policy accidentally treating Sleep as an error-like continuation.

## Responsibilities By Layer

### `crates/core`

No behavioral ownership changes in this slice.

- `SleepTool` stays responsible for:
  - validating `duration_ms`
  - clamping to `[60_000, 86_400_000]`
  - cache-aware snapping when enabled
  - returning the sentinel payload

`core` does not schedule timers and does not know goal lifecycle state.

### `crates/driver-types`

Add the typed `AttemptOutcome::Sleep { duration_ms, reason }` variant.

Why:

- separates "explicit park the goal" from generic continue/retry semantics
- gives `orchestrator.rs` a clean, auditable match arm
- creates the right seam for `77.20.3` persistent `Sleeping { wake_at, reason }`

### `crates/driver-loop/src/attempt.rs`

`attempt.rs` remains the owner of one Claude turn.

New responsibility:

- recognize when the turn result corresponds to a `Sleep` sentinel
- translate it into `AttemptOutcome::Sleep`
- avoid surfacing the sentinel as ordinary `final_text` or ordinary follow-up result handling

`attempt.rs` must not grow timer logic.

### `crates/driver-loop/src/orchestrator.rs`

`orchestrator.rs` remains the owner of goal lifecycle.

New responsibility:

- branch on `AttemptOutcome::Sleep`
- compute the wake deadline from `duration_ms`
- park the goal in-process
- await wake-up with cancellation awareness
- populate `next_extras` with a synthetic `<tick>` payload
- continue the loop without going through replay-policy

### `crates/driver-loop/src/proactive.rs` (new)

Create a focused module for proactive runtime helpers.

Planned contents:

- `ScheduledWake` struct containing:
  - `duration_ms`
  - `reason`
  - `sleep_started_at`
- helper to wait until wake-up with `tokio::select!`
- helper to construct stable tick extras / tick text

This is the "dedicated scheduler" for the slice: a dedicated module and control path, not a separate service.

## Control Flow

### Normal Sleep Path

1. A normal proactive turn starts.
2. The model calls `Sleep { duration_ms, reason }`.
3. `SleepTool` returns the sentinel payload.
4. `attempt.rs` detects the sentinel and returns:

```rust
AttemptOutcome::Sleep {
    duration_ms,
    reason,
}
```

5. `orchestrator.rs` receives the attempt result.
6. The orchestrator does **not** pass this through replay-policy.
7. The orchestrator creates a `ScheduledWake` and waits.
8. When the timer fires, the orchestrator builds `next_extras` for the next turn.
9. The loop continues and the next attempt receives a synthetic `<tick>` prompt.

### Tick Injection Shape

The wake-up prompt should be short, deterministic, and meta.

Proposed shape:

```text
<tick>
kind: sleep_wake
elapsed_ms: 270000
reason: waiting for new work
</tick>
```

Guidelines:

- no long prose
- no unstable timestamp formatting inside the visible prompt
- no duplicated narration from the runtime
- enough structure for the model to reason about why it woke up

The visible prompt content should come from one helper so formatting cannot drift across code paths.

### Prompt Construction

The tick block is prepended to the next turn in the same place that currently prepends operator interrupts and compact commands.

That keeps one prompt assembly choke-point in `attempt.rs` and avoids scattering prompt composition across modules.

## Cancellation And Pause Semantics

### Cancellation During Sleep

If `cancel_goal()` or orchestrator shutdown happens while waiting:

- the wake timer must stop immediately
- the goal exits as `AttemptOutcome::Cancelled` at orchestrator level
- no synthetic tick is injected afterward

This is implemented with a cancellation-aware wait using `tokio::select!`.

### Pause During Sleep

If `pause_goal()` is invoked while the goal is sleeping:

- the sleep timer itself does not need to pause
- after wake-up, the existing pause gate at the top of the main loop remains authoritative

This preserves current pause semantics and avoids inventing a second pause model for timers.

### Repeated Sleep Cycles

Multiple `Sleep -> wake -> Sleep` cycles in the same process must be supported. The orchestrator path must be idempotent per cycle and must not leak stale `next_extras` between turns.

## Non-Goals

This slice deliberately does **not** solve:

- durable registry state transitions such as `Running -> Sleeping -> Running`
- wake-up triggered by real user/channel input
- proactive daily turn budgets
- proactive metrics counters

Keeping those out prevents `77.20.2` from turning into a cross-cutting persistence/ops change.

## Error Handling

### Invalid Duration

`SleepTool` already clamps invalid durations. `driver-loop` should trust the effective `duration_ms` it receives and must not re-clamp with divergent rules.

### Missing Reason

`SleepTool` already defaults missing reason text. `driver-loop` should treat the reason as opaque text and pass it through unchanged.

### Stream Edge Cases

If the Claude turn ends without a proper Sleep sentinel, existing handling remains unchanged.

Only a positively identified sentinel may produce `AttemptOutcome::Sleep`.

## Testing Strategy

### Unit Tests

1. Tick payload formatting is stable and includes:
   - `kind`
   - `elapsed_ms`
   - `reason`
2. Cancellation-aware wait exits immediately when cancelled.
3. Repeated scheduling does not leak state between wakes.

### Integration Test

Add a `driver-loop` integration test that simulates:

1. first turn returns `Sleep`
2. orchestrator waits
3. second turn receives the synthetic `<tick>`
4. second turn does real work or exits cleanly

Assertions:

- the goal does not terminate on the first `Sleep`
- the wake-up path is taken exactly once
- the next attempt sees the tick payload
- replay-policy is not invoked for the Sleep path

## Acceptance Criteria For This Slice

`77.20.2` is complete when:

- `Sleep` never re-enters the model as a normal tool result
- the orchestrator can park an in-process goal and wake it later
- wake-up happens through a deterministic `<tick>` prompt
- cancellation during sleep is safe
- the implementation leaves a clean seam for `77.20.3` persistence

## Deferred Follow-Up Into `77.20.3`

The next slice should persist the same conceptual state introduced here:

- sleep reason
- wake deadline
- proactive-enabled resume semantics

That persistence slice should reuse the same typed control-flow rather than reinterpret raw sentinel payloads again.
